# پنل مدیریت کاربران و کانفیگ

اپلیکیشن وب کامل (بک‌اند Node.js/Express + دیتابیس SQLite + فرانت‌اند) برای مدیریت کاربران و تولید کانفیگ اتصال (VLESS / VMess).

## پیش‌نیاز
- Node.js نسخه ۱۸ به بالا

## نصب و اجرا

```bash
cd config-panel
npm install
cp .env.example .env
```

فایل `.env` رو باز کن و این مقادیر رو عوض کن:
- `JWT_SECRET`: یک رشته‌ی تصادفی و طولانی
- `ADMIN_USERNAME` / `ADMIN_PASSWORD`: مشخصات ورود اولیه‌ی ادمین (فقط در اولین اجرا استفاده می‌شه و در دیتابیس هش می‌شه)

بعد اجرا کن:

```bash
npm start
```

پنل روی آدرس `http://localhost:3000` بالا میاد (یا پورتی که در `.env` تنظیم کردی).

## دیپلوی روی Railway

۱. کد رو به یه ریپازیتوری GitHub پوش کن (Railway از GitHub دیپلوی می‌کنه)
۲. توی railway.app یه پروژه‌ی جدید بساز و گزینه‌ی "Deploy from GitHub repo" رو انتخاب کن
۳. توی تنظیمات پروژه (Variables) این‌ها رو اضافه کن:
   - `JWT_SECRET` = یک رشته‌ی تصادفی طولانی
   - `ADMIN_USERNAME` = نام کاربری دلخواه
   - `ADMIN_PASSWORD` = رمز قوی
   - `DB_PATH` = `/data/data.sqlite` (مسیر دیتابیس روی ولوم)
   (نیازی به تنظیم `PORT` نیست، Railway خودش می‌ده)
۴. یک **Volume** به پروژه اضافه کن (تب Settings یا Volumes) و مسیر Mount رو `/data` بذار — این کار لازمه چون فایل‌سیستم Railway بدون Volume در هر دیپلوی جدید پاک می‌شه و دیتابیست از بین می‌ره
۵. بعد از دیپلوی موفق، از تب Settings پروژه گزینه‌ی "Custom Domain" رو بزن و دامنه‌ی خودت رو اضافه کن، بعد یک رکورد CNAME توی DNS دامنه‌ت به آدرسی که Railway می‌ده اضافه کن

## دیپلوی روی سرور واقعی
- می‌تونی با `pm2` پروسه رو دائمی نگه داری: `pm2 start server.js --name config-panel`
- پشت Nginx بذارش و با TLS (Let's Encrypt / Certbot) روی دامنه‌ی خودت سرو کن
- فایل `data.sqlite` همون دیتابیسه — بکاپ‌گیری منظم ازش رو فراموش نکن

## نکته‌ی مهم درباره‌ی کانفیگ‌ها (سرور داخلی Railway)

این پنل الان یک نسخه‌ی **Xray-core** رو هم داخل همون کانتینر Railway اجرا می‌کنه — یعنی دیگه نیاز به یک VPS جدا نداری. جزئیات فنی:

- روی Railway از **Dockerfile** پروژه استفاده می‌شه (نه Nixpacks پیش‌فرض)، چون در زمان build باید باینری Xray-core دانلود بشه.
- چون Railway فقط ترافیک HTTP(S) رو (روی همون دامنه‌ی `*.up.railway.app`) به کانتینر پاس می‌ده، فقط پروتکل **VLESS + WebSocket** واقعاً کار می‌کنه. سایر گزینه‌های پروتکل (gRPC، TCP خام، VMess، Shadowsocks) فقط وقتی کار می‌کنن که یه سرور Xray/V2Ray جدا (روی یک VPS واقعی) داشته باشی.
- پورت همیشه ۴۴۳ و Security همیشه TLS هست، چون TLS رو خود Railway در لبه‌ی شبکه‌ش انجام می‌ده و بعد ترافیک ساده (بدون رمزنگاری اضافه) رو به اپ می‌فرسته.
- هر بار که یه کاربر بسازی/ویرایش/حذف/ریست کنی، پنل خودش فایل تنظیمات Xray رو دوباره می‌سازه و پروسه‌ی Xray رو ری‌استارت می‌کنه — دستی نیاز به کاری نیست.
- **محدودیت دستگاه** (`device_limit`) فعلاً فقط جنبه‌ی نمایشی/اطلاعاتی داره و توسط Xray به‌صورت خودکار اعمال نمی‌شه (این کار نیاز به یک ماژول کنترل هم‌زمانی جدا داره).
- اگه بخوای از یک VPS واقعی هم استفاده کنی (برای پروتکل‌های دیگه یا کارایی بهتر)، همون قابلیت قبلی (فقط تولید لینک کانفیگ بدون اجرای سرور) رو می‌تونی برای اون کاربرا جدا نگه داری.

### چرا حتماً باید Dockerfile استفاده بشه؟
اگه Railway به‌جای Dockerfile از Nixpacks استفاده کنه، باینری Xray نصب نمی‌شه و کانفیگ‌ها فقط لینک تولید می‌کنن ولی وصل نمی‌شن. فایل `railway.json` این پروژه از قبل Railway رو مجبور می‌کنه از Dockerfile استفاده کنه؛ لازم نیست کاری دستی انجام بدی، فقط مطمئن شو این فایل حذف نشده باشه.

## ساختار پروژه
```
config-panel/
  server.js            # نقطه‌ی ورود Express + پراکسی WebSocket به Xray
  Dockerfile            # نصب Xray-core در زمان build روی Railway
  railway.json          # اجبار Railway به استفاده از Dockerfile
  lib/configBuilder.js   # ساخت رشته‌ی اتصال (vless://, vmess://, ss://)
  lib/xrayManager.js     # تولید config.json و مدیریت پروسه‌ی Xray
  db/index.js            # اتصال SQLite و schema
  routes/auth.js         # لاگین و تغییر رمز
  routes/users.js        # CRUD کاربران + تولید کانفیگ
  routes/settings.js     # تنظیمات سرور
  routes/subscription.js # لینک ساب عمومی (/sub/:uuid)
  public/                # فرانت‌اند (HTML/CSS/JS ساده)
```
